## Distribution and Packaging Options  
We want **easy installation** with minimal setup.  In Python this usually means publishing to PyPI and installing via `pip`.  However, end-user CLI tools often use **pipx**: unlike `pip`, which installs into the active Python environment, pipx creates an isolated environment for each app but exposes its commands globally【55†L146-L154】【55†L261-L270】.  In practice, this means users can `pipx install vibecheck` (or `pip install vibecheck`) and run `vibecheck` anywhere without dependency conflicts.  Alternatively, packaging for Homebrew or apt could be considered (as one author did with a CLI tool)【60†L158-L166】.  The Dirmapper example shows that migrating to a modern `pyproject.toml` with the correct `src/` layout and entry point is key to making a CLI installable via PyPI【60†L158-L166】.  In summary, we should structure vibecheck as a standard Python package (with scripts in `pyproject.toml`) and recommend users install it (globally or per-venv) with pip or pipx.  We can support both “repo-scoped” (e.g. a dev dependency or `python -m vibecheck`) and global usage via pipx; pipx is often ideal for end-user tools【55†L146-L154】【55†L261-L270】.

## Local-First Architecture and OpenRouter Integration  
We aim for a **local-first** design: all core data and processing happen on the user’s machine, with only LLM calls going to the cloud.  This minimizes trust and privacy concerns.  For example, the *RepoReviewer* system (arXiv 2026) is explicitly built as a local-first review tool with CLI, local API, and web dashboard, writing outputs to disk rather than relying on servers【9†L13-L20】.  Similarly, Plannotator (a plan-review tool) “runs entirely locally” with no backend or network traffic, ensuring plans never leave the user’s machine【5†L78-L87】. We should adopt this philosophy: store the competence model and config in local files (e.g. JSON/TOML under `.vibecheck/`), and only send minimal prompts to LLMs. 

For LLM access, we plan to use **OpenRouter** as a provider-agnostic gateway. OpenRouter exposes a single unified API to access many models (OpenAI, Anthropic, Google, etc.)【39†L176-L184】.  In practice, one can even use standard OpenAI SDK calls pointed to the OpenRouter base URL. For example, the Snyk tutorial shows creating an `OpenAI` client with `base_url="https://openrouter.ai/api/v1"`, which “works because OpenRouter maintains full compatibility with OpenAI’s API format”【39†L281-L289】. This means our code can switch providers by reconfiguring the base URL and API key.  We should provide configuration (or a CLI auth command) for the user’s OpenRouter key; for example, as one tool does we might store the API key in `~/.vibecheck/.env` (file permission 0600) for security【62†L547-L548】.  Overall, the integration with OpenRouter should be straightforward and keep the system provider-agnostic【39†L281-L289】.

## Interactive Plan-Review Tools (Plannotator and Peers)  
To give developers insight into AI plans, we can take inspiration from tools like **Plannotator**. Plannotator provides a browser UI for reviewing an agent’s plan: users can select parts of a plan and mark them for deletion, add comments, or suggest edits【5†L47-L55】. It then generates a modified plan accordingly. Critically, Plannotator “runs entirely locally” and integrates natively with coding agents【6†L9-L14】【5†L78-L87】. This aligns with our goal: we can offer a Gradio-based UI where the developer sees the initial competence-plan, and can annotate or refine it.  For example, Plannotator’s steps (“click to select, mark for deletion, add comment, and update the plan”) can guide our design【5†L47-L55】【6†L9-L14】.  Users would interact in the browser, but all data stays on-device. Integrating such interactive plan review can help validate our competence model’s recommendations.

## Competence Model Data Structures  
The competence model will capture *concept nodes* and their relationships. A natural approach is to use a **graph** or **knowledge graph** to store explicit concept relationships, possibly augmented by a **vector database** for similarity matching.  For example, a graph database (like Neo4j or TigerGraph) could hold nodes for concepts and edges representing prerequisite or thematic links. A knowledge graph preserves context and reasoning paths by explicitly connecting entities【46†L193-L202】.  Meanwhile, each concept’s description can be embedded (via an LLM) and indexed in a vector DB (such as Pinecone or Weaviate). Vector databases excel at semantic similarity: given a new phrase, we can retrieve the closest concept nodes by nearest-neighbor search【46†L193-L202】. In practice we could do hybrid queries: use vector search to find candidate concepts and graph traversals to validate or aggregate evidence.  The PuppyGraph blog notes that knowledge graphs answer “How is A connected to B?”, while vector DB answers “Find items similar to X”【46†L193-L202】.  We likely need both: a graph (or RDF store) for the structured concept network, and a vector index for adding or matching semantically similar concepts. This dual approach is common in retrieval-augmented systems【46†L193-L202】.

## Decision Algorithms for Competence Aggregation  
We must decide how to update and combine evidence about a developer’s skill. One approach is **probabilistic reasoning** (Bayesian models). In educational tech, *Bayesian Knowledge Tracing* (BKT) models a learner’s mastery of concepts as latent states in a Markov model【57†L39-L44】. Analogously, we could maintain a probability or score for each concept node indicating the developer’s competency. Each survey response or feedback is evidence to update these probabilities. The KT literature emphasizes that Bayesian models estimate *latent knowledge states* from observed interactions【57†L39-L44】【50†L123-L131】. For example, a correct answer might increase belief in mastery of a concept, while an error would decrease it, in a principled way.

We can also consider **graph search or planning algorithms**. If our concept network is weighted (e.g. “distance” to target skill), algorithms like **A\*** could find an efficient path from a user’s current knowledge state to a goal node【52†L51-L59】. A* works by minimizing `f = g + h`, where *g* is cost so far and *h* is a heuristic to the goal【52†L51-L59】. For instance, if we want to chart a learning path, A* could prioritize concepts that bridge the largest knowledge gap.  Finally, a **MDP/POMDP** framework could model the sequential decision of which concept to quiz or teach next, optimizing a reward (improved competency). In sum, Bayesian updating (for inference) combined with heuristic search (for planning) are promising strategies for competence aggregation.

## Demo and Initialization Plan  
To demonstrate the system, we’ll need both a CLI and a simple UI. We propose a command like `vibecheck cm init` that launches an interactive survey (e.g. a Gradio app).  In that web UI, the user rates their skill level in a few key concepts (seed values), which populates initial nodes in the competence model. Based on their answers, the backend initializes concept nodes (creating new ones or linking to existing ones if they match semantically).  The UI could present the preliminary competence graph or hierarchy and allow the user to add or merge concepts on the fly.  For example, if a user enters a new topic, the system might use vector similarity to decide if it’s essentially the same as an existing node or a genuinely new concept. 

We also plan a CLI command like `vibecheck auth` to configure the LLM provider. Rather than requiring the user to manually edit environment vars, the CLI can prompt for the OpenRouter API key (and store it in `~/.vibecheck/.env` with file perms 0600)【62†L547-L548】. (This mirrors how other tools securely store API keys locally.) Finally, if we target integration with Claude Code (Anthropic’s CLI), we might provide a `vibecheck cc init` or plugin to bootstrap the tool within that environment. Overall, the demo would show installing vibecheck (via pip/pipx), running `vibecheck cm init` to set up competencies, and then having an LLM-driven gate or plan using those competencies.
